Verifica Pratica · Informatica

Server REST
Roulette

Implementazione di un servizio web RESTful per la gestione delle vincite alla roulette, tramite richieste GET e POST.

Progetto NetBeans ServerRest-Verifica
Endpoint base /api/roulette/paridispari
Porta locale 8080

01 — Obiettivo

Cosa devi realizzare

Il progetto NetBeans ServerRest-Verifica contiene già lo scheletro dell'applicazione. Il tuo compito è completare l'implementazione delle classi e dei metodi necessari affinché il server sia in grado di ricevere una richiesta, analizzarla e restituire al client la risposta corretta in formato JSON.

In sintesi, il server deve ricevere due informazioni — la puntata del giocatore e il numero estratto dalla roulette — e decidere se il giocatore ha vinto oppure no, restituendo il risultato strutturato come oggetto JSON.

⚑ Attenzione

Non modificare la struttura del progetto né i file già predisposti, se non nei punti indicati come da completare. Lavora esclusivamente all'interno del progetto ServerRest-Verifica.

02 — Logica di Gioco

Le regole della Roulette

La logica che determina la vittoria dipende esclusivamente dal numero estratto. Esistono due possibili esiti, ciascuno legato a una condizione matematica sul numero:

R

DISPARI

Vince se il numero estratto è dispari (e quindi diverso da 0).

N

PARI

Vince se il numero estratto è pari e diverso da 0.

03 — Parametri

I dati in ingresso

Il server accetta due parametri obbligatori, sia nelle richieste GET (come parametri nella URL) sia nelle richieste POST (come campi del payload JSON):

Parametro Tipo Valori accettati Descrizione
giocata String "DISPARI" | "PARI" La puntata effettuata dal giocatore
numero String (intero) "0" … "36" Il numero estratto dalla roulette
Conversione del numero

Il parametro numero arriva sempre come stringa. Dovrai convertirlo in intero per poter applicare la logica di parità/disparità.

04 — Metodo GET

Richiesta GET

L'endpoint GET si trova al percorso /api/roulette/paridispari/get. I parametri vengono passati direttamente nella URL tramite la sintassi query string (il punto interrogativo seguito da coppie chiave=valore separate da &).

Esempi di URL valide

GET Puntata sul DISPARI — numero 12
URL della richiesta 
http://localhost:8080/api/roulette/paridispari/get?giocata=DISPARI&numero=12
Risposta JSON attesa 
{
  "giocata"  : "DISPARI",
  "numero"   : 12,
  "vittoria" : false   // 12 è pari → il DISPARI perde
}
GET Puntata sul PARI — numero 12
URL della richiesta 
http://localhost:8080/api/roulette/paridispari/get?giocata=PARI&numero=12
Risposta JSON attesa 
{
  "giocata"  : "PARI",
  "numero"   : 12,
  "vittoria" : true    // 12 è pari → il PARI vince
}
GET Puntata sul DISPARI — numero 0 (caso speciale)
URL della richiesta 
http://localhost:8080/api/roulette/paridispari/get?giocata=DISPARI&numero=0
Risposta JSON attesa 
{
  "giocata"  : "DISPARI",
  "numero"   : 0,
  "vittoria" : false   // 0 non è dispari → il DISPARI perde
}

05 — Metodo POST

Richiesta POST

L'endpoint POST si trova al percorso /api/roulette/paridispari/post. In questo caso i parametri non vengono inseriti nella URL, ma inviati nel corpo della richiesta (body) come oggetto JSON, con Content-Type application/json.

Struttura del payload JSON

{
  "giocata" : "DISPARI",
  "numero"  : 12
}

Esempi completi POST

POST Puntata sul DISPARI — numero 7
URL 
http://localhost:8080/api/roulette/paridispari/post
Body (JSON inviato) 
{
  "giocata" : "DISPARI",
  "numero"  : 7
}
Risposta JSON attesa 
{
  "giocata"  : "DISPARI",
  "numero"   : 7,
  "vittoria" : true    // 7 è dispari → il DISPARI vince
}

06 — Risposta

Struttura della risposta JSON

Qualsiasi richiesta — GET o POST — deve sempre produrre una risposta JSON con esattamente tre campi, nell'ordine indicato di seguito: 

{
  "giocata"  : "<la puntata ricevuta>",
  "numero"   : <il numero ricevuto>,
  "vittoria" : true oppure false
}

Il campo vittoria deve essere restituito come stringa contenente il valore booleano ("true" o "false"), non come valore booleano nativo JSON. Tutti e tre i valori devono essere restituiti esattamente come ricevuti (giocata e numero) o calcolati (vittoria).

Tabella di verità rapida

DISPARI + dispari (≠0) → true  |  DISPARI + pari o 0 → false
PARI  + pari (≠0) → true  |  PARI  + dispari o 0 → false

Sezione Avanzata
Parte II
API v2 — Colori

La versione 2 delle API introduce la logica reale della roulette: i numeri rossi e neri non seguono più la semplice regola pari/dispari, ma corrispondono alla distribuzione autentica del tavolo da gioco. L'endpoint cambia da paridispari a colori.

A — Logica di Gioco v2

Le regole reali della Roulette

Nella versione 2, la vittoria non dipende più dalla parità del numero, ma dall'appartenenza del numero estratto ai due insiemi fissi del tavolo da roulette. I numeri rossi e neri sono distribuiti in modo asimmetrico e non corrispondono alla semplice alternanza pari/dispari.

R

ROSSO

Vince se il numero estratto appartiene all'insieme dei 18 numeri rossi della roulette.

N

NERO

Vince se il numero estratto appartiene all'insieme dei 18 numeri neri della roulette.

Insiemi di numeri

I numeri associati a ciascun colore sono i seguenti. Il server dovrà verificare l'appartenenza del numero estratto all'insieme corrispondente alla giocata ricevuta.

Numeri Rossi (18)
1 3 5 7 9 12 14 16 18 19 21 23 25 27 30 32 34 36
Numeri Neri (18)
2 4 6 8 10 11 13 15 17 20 22 24 26 28 29 31 33 35
⚑ Differenza chiave rispetto alla v1

In v1 il numero 12 fa vincere il PARI (è pari). In v2, invece, il 12 è un numero ROSSO nella roulette reale, quindi fa vincere chi ha puntato su ROSSO e perdere chi ha puntato su NERO. La logica non è più matematica ma si basa su una ricerca nell'insieme dei numeri del proprio colore.

B — Endpoint v2

URL delle API v2

Gli endpoint della versione 2 hanno la stessa struttura della v1, con l'unica differenza nel segmento del percorso: paridispari diventa colori. Rispetto alla v1, la v2 introduce un nuovo parametro obbligatorio in ingresso (importo) e due nuovi campi nella risposta (importogiocato e importoriscosso).

Metodo Endpoint v2 Note
GET /api/v2/roulette/colori/get Parametri in query string
POST /api/v2/roulette/colori/post Parametri nel body JSON

Parametri v2

Parametro Tipo Dove Descrizione
giocata String GET & POST Il colore puntato: "ROSSO" | "NERO"
numero String (intero) GET & POST Il numero estratto dalla roulette: "0" … "36"
importo float GET & POST L'importo della giocata in euro, con decimali (es. 50.00)

Campi della risposta v2

Campo Tipo Descrizione
giocata String Il colore puntato, restituito così com'è
numero int Il numero estratto, restituito così com'è
vittoria boolean Esito della giocata: true oppure false
importogiocato float L'importo ricevuto in ingresso, restituito invariato
importoriscosso float In caso di vittoria: importo × 2  |  In caso di sconfitta: 0.0

E — Riepilogo v2

Logica di verifica

L'implementazione del metodo di controllo dovrà effettuare una ricerca per appartenenza all'interno di due strutture dati che contengono i numeri di ciascun colore, anziché un calcolo matematico di parità. Il numero ricevuto come stringa va prima convertito in intero, poi cercato nell'insieme del colore puntato.

Tabella di verità v2

ROSSO + numero ∈ {1,3,5,7,9,12,14,16,18,19,21,23,25,27,30,32,34,36} → vittoria true
ROSSO + qualsiasi altro numero → vittoria false
NERO  + numero ∈ {2,4,6,8,10,11,13,15,17,20,22,24,26,28,29,31,33,35} → vittoria true
NERO  + qualsiasi altro numero → vittoria false

importogiocato = valore di importo ricevuto in ingresso (invariato)
importoriscosso = importo × 2 se vittoria true  |  0.0 se vittoria false
